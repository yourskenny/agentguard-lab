## 正文草稿

### 1. 创意名称 + 创意介绍

创意名称：AgentGuard Lab：AI Agent 可靠性评测与修复工作台。

我想做的是一个面向 AI Agent 开发者的评测与修复工作台。现在很多学生和团队都能快速做出课程助手、知识库问答助手、代码评审助手或营销助手，但这些 Agent 往往只展示顺利样例，缺少系统化的可靠性测试。一旦遇到资料不足、提示词冲突、工具参数错误、越权问题或上下文污染，开发者很难判断问题出在哪里，也很难证明修改后真的变好了。

AgentGuard Lab 的核心思路是：用户输入一个 Agent 的系统提示词、知识库片段、工具说明或对话记录，系统自动生成评测用例，运行事实性、边界拒答、工具调用、上下文污染、幻觉和指令冲突等测试，定位失败原因，并给出可复测的修复建议。它不是一个普通聊天助手，而是帮助 Agent 项目从“能演示”走向“可测试、可解释、可维护、可交付”的质量保障工具。

产品形态计划做成一个网页工作台：左侧选择 Agent 样例或上传配置，中间运行评测，右侧查看失败归因和修复建议，底部展示修复前后的通过率对比和导出报告。

### 2. 目标用户及痛点

目标用户包括：正在做 AI Agent 课程项目、竞赛项目、开源项目或内部工具的学生开发者和小团队。他们通常已经能做出一个基本 Demo，但很难进一步回答这些问题：这个 Agent 到底稳定不稳定？它会不会编造事实？会不会泄露不该输出的信息？工具调用失败时能不能正确处理？修改提示词或知识库后，效果是否真的提升？

如果没有 AgentGuard Lab，开发者通常只能靠人工试问几轮，或者挑选成功样例进行展示。这种方式容易遗漏失败场景，也无法形成可复用的回归测试。对于比赛、答辩或项目交付来说，仅仅展示“能聊”已经不够，更重要的是证明这个 Agent 有明确边界、有失败处理能力、有可复测的质量改进过程。

### 3. 价值与意义

AgentGuard Lab 的价值在于把 AI Agent 开发中的质量验证流程产品化。它能帮助开发者自动构造测试集、发现失败样例、解释失败原因，并输出下一轮修复建议，让 Agent 项目拥有更接近真实工程交付的质量闭环。

对学生开发者来说，它可以提升课程项目、竞赛项目和作品集项目的可信度；对 AI 应用团队来说，它可以作为轻量级 Agent 回归测试工具；对评审或导师来说，它也能让一个 AI Agent 的可靠性不再只靠演示印象，而是有测试结果、失败证据和复测对比可看。
